1. Resumo executivo

A base atual já entrega um editor didático de lógica digital muito competente: portas básicas, blocos combinacionais nativos, tabela verdade, simulação sequencial nativa, simulação com realimentação usando estado anterior dos sinais, exemplos e testes automatizados do core.

Diagnóstico: o comportamento principal está validado e a experiência visual está boa. O próximo trabalho não deve ser reescrever, mas extrair módulos estáveis para consolidar um framework interno.

O maior risco hoje é concentração de responsabilidades em poucos arquivos: App.tsx tem cerca de 1457 linhas, CircuitCanvas.tsx cerca de 1083 linhas e evaluateCircuit.ts concentra simulação combinacional, sequencial nativa, realimentação e helpers de estado.

2. O que o projeto é hoje

Editor visual SVG
Canvas com pan, zoom, seleção, drag, múltipla seleção, menu contextual, edição de nomes, resize de texto e conexões por pinos.
Simulador combinacional
Portas, blocos combinacionais nativos e tabela verdade do circuito inteiro para circuitos sem memória/feedback.
Sequencial nativo
Clock, Latch D, Flip-Flop D, botão Tick e painel de estado sequencial.
Realimentação emergente
Latches feitos com NOR/NAND comuns funcionam porque o simulador usa o último estado dos sinais como ponto de partida.
Didática
Exemplos com textos explicativos, arquivos JSON para testes manuais e documentação HTML para próximos devs.
Persistência
Autosave em localStorage, import/export JSON e preferência de estilo de fio.

4. Modelo de dados atual

interface CircuitDocument {
  version: 1;
  components: LogicComponent[];
  wires: Wire[];
}

interface LogicComponent {
  id: string;
  type: GateType;
  x: number;
  y: number;
  label?: string;
  state?: boolean;
  width?: number;
  memory?: Record<string, boolean>;
}

O modelo é simples e produtivo. Ele funcionou bem para o MVP. O campo memory está sendo usado por componentes sequenciais nativos, enquanto a memória emergente de realimentação fica no SimulationState.values, mantido no runtime pela UI.

Ponto de decisão: consolidar a diferença entre estado do documento e estado da simulação. Hoje parte está no documento (component.memory) e parte em runtime (SimulationState em App).

Recomendação

Manter CircuitDocument como estrutura persistente e criar um módulo explícito de runtime. O memory dos componentes pode continuar temporariamente por compatibilidade, mas o framework deve caminhar para: 

interface SimulationRuntime {
  values: EvaluationResult;
  componentMemory: Record<string, Record<string, boolean>>;
  unstable: boolean;
  iterations: number;
}

5. Estado da simulação

O core atual tem uma ideia importante e correta:

Portas continuam puras; o simulador lembra sinais.
Isso permite que latches feitos com portas comuns mantenham estado sem transformar AND/NOR/NAND conceitualmente em componentes sequenciais.

Como funciona hoje

  1. simulateCircuit(circuit, previousState?) inicializa pinos com valores anteriores quando disponíveis.
  2. Inputs, buttons, clock e Q de sequenciais sobrescrevem seus valores fonte.
  3. Componentes são avaliados iterativamente.
  4. Fios propagam valores.
  5. Se nada muda, estabilizou; se exceder MAX_ITERATIONS = 64, marca unstable.

Riscos técnicos

  • evaluateCircuit() sem estado anterior ainda existe como wrapper; em circuitos com feedback pode não representar a sessão real.
  • settleSequentialCircuit() e stepCircuit() ainda misturam lógica de runtime com alteração de documento.
  • inputValue() busca fios com find(); em circuitos grandes, isso vira custo repetido e dificulta raciocínio.
  • A detecção de instabilidade é por limite de iteração, boa para MVP, mas ainda não distingue oscilação real de convergência lenta.

Separação sugerida

src/core/simulation/
  signals.ts          // signalKey, read/write, clone, inicialização
  gates.ts            // evaluatePrimitiveGate, evaluateNativeBlock
  simulate.ts         // simulateUntilStable
  sequential.ts       // clock, d-latch, d-flip-flop, stepNativeSequential
  graph.ts            // feedback/cycle detection
  truthTable.ts       // geração de tabela verdade

7. Testes e validação

Já existe npm test, usando Vite para gerar um bundle Node do teste TypeScript. Os testes atuais cobrem:

  • XOR combinacional básico;
  • Latch SR com NOR mantendo estado;
  • Latch SR com NAND ativo baixo mantendo estado;
  • Latch D construído com NANDs;
  • NOT realimentado detectado como instável;
  • checagem básica de Flip-Flop D nativo.
Importante: esses testes são o cinto de segurança para a refatoração do core. Antes de mover simulação para módulos, rode npm test e mantenha esses casos verdes.

Testes que faltam

  • stepCircuit() diretamente: borda de subida, borda de descida e múltiplos ticks.
  • Tabela verdade: circuito combinacional com múltiplas entradas/saídas.
  • Feedback detector: ciclos simples, ciclos entre múltiplos componentes e circuitos acíclicos.
  • Import/export/normalização: componentes antigos sem memory.
  • Roteamento puro: self-loop ortogonal e curvo, colisões e jumps.

9. Framework interno desejado

O objetivo da refatoração deve ser transformar o projeto em camadas claras:

Domain Model
  CircuitDocument, LogicComponent, Wire, PinRef

Catalog
  component definitions, pins, sizes, labels

Simulation Engine
  pure gate evaluation, signal runtime, feedback settling, sequential stepping

Editor State
  history, selection, tools, import/export, simulation runtime bridge

Canvas Renderer
  SVG rendering, wire routing, interaction hooks

Educational Shell
  examples, truth table, sequential panel, annotations

API alvo do core

const runtime = createSimulationRuntime(circuit);
const settled = simulateUntilStable(circuit, runtime);
const afterInput = applyInputChange(circuit, settled.runtime, inputId, true);
const afterTick = stepSimulation(circuit, afterInput.runtime);
const truth = buildTruthTable(circuit);

A UI deveria chamar ações de alto nível. Ela não deveria saber detalhes como quando usar estado anterior, quando normalizar memória ou como detectar feedback.

10. Plano de refatoração incremental

Fase 0 — Congelar comportamento

  1. Rodar npm test e npm run build.
  2. Adicionar testes diretos para stepCircuit().
  3. Adicionar teste para circuitHasFeedback() antes de extrair.

Fase 1 — Extrair exemplos e painéis

  1. Mover CIRCUIT_EXAMPLES para src/examples/circuitExamples.ts.
  2. Mover CircuitTruthTable e helpers para src/ui/panels/CircuitTruthTable.tsx.
  3. Mover SequentialStatePanel para src/ui/panels/SequentialStatePanel.tsx.

Risco baixo. Deve reduzir bastante App.tsx sem alterar o core.

Fase 2 — Extrair simulação sem mudar API

  1. Criar src/core/simulation/signals.ts com read/write/clone/init.
  2. Criar src/core/simulation/gates.ts com lógica de componentes puros.
  3. Criar src/core/simulation/simulate.ts com simulateCircuit.
  4. Manter evaluateCircuit.ts como fachada/export compatível temporária.

Risco médio. Protegido pelos testes de latch/feedback.

Fase 3 — Runtime explícito no App

  1. Substituir simulationStateRef por hook useSimulationRuntime(circuit).
  2. Adicionar botão Resetar simulação.
  3. Definir quando runtime é preservado ou reiniciado: importar, carregar exemplo, undo/redo, remover componentes.

Fase 4 — Extrair canvas

  1. Mover roteamento para src/ui/editor/wireRouting.ts.
  2. Mover WireView, ComponentView e PendingWire para renderers.
  3. Criar hooks para câmera, seleção e drag.

Fase 5 — Crescimento controlado

  1. Clock automático com play/pause.
  2. Registradores e contadores.
  3. Subcircuitos reais, se a arquitetura estiver estável.

11. Contratos que a refatoração deve preservar

  • JSON versão 1 deve continuar abrindo.
  • Circuitos combinacionais sem feedback devem continuar com tabela verdade.
  • Circuitos com feedback devem preservar estado usando valores anteriores.
  • NOT realimentado deve ser marcado como instável.
  • Latch SR NOR/NAND feito com portas comuns deve manter estado.
  • Latch D nativo e feito com portas devem continuar funcionando.
  • Uma entrada deve aceitar somente um fio; uma saída pode alimentar várias entradas.
  • Self-loop deve ser permitido e visualmente claro.
  • Undo/redo não deve explodir histórico durante drag/resize.
  • Import/export/autosave devem continuar funcionando.

12. Checklist prático para cada etapa

Antes de editar
  • Rodar npm test.
  • Rodar npm run build.
  • Escolher uma refatoração pequena.
Durante
  • Manter APIs antigas como fachadas.
  • Não misturar mudança visual com mudança de core.
  • Evitar reformatar arquivos enormes sem necessidade.
Depois
  • Rodar npm test.
  • Rodar npm run build.
  • Testar manualmente um latch SR e um circuito combinacional.
Conclusão: a base está madura para uma refatoração incremental. O core conceitual certo já existe; agora o objetivo é dar nomes, fronteiras e testes suficientes para que o projeto possa crescer como framework.
